# 大厂的 Git 代码规范

## 📝 概要

Git 代码规范是一种帮助团队协作、高效管理代码的最佳实践，特别是在多人开发的大型项目中。规范化的分支命名、提交流程和版本管理可以提高开发效率，减少冲突与错误。

## 📌 核心知识点

### **1. Git 分支命名规范**

- **master**：主分支，稳定的生产版本。

- **develop**：开发分支，合并所有开发功能。

- **feature**：开发新功能，基于 `develop` 创建，命名如 `feature/xxx`。

- **test**：测试环境，确保在此分支上测试通过。

- **release**：发布准备分支，准备上线的版本。

- **hotfix**：热修复分支，修复紧急 bug。

### **2. 分支与环境对应关系**

|分支|功能|环境|
|---|---|---|
|master|主分支，稳定版|PRO（生产）|
|develop|开发分支，最新版本|DEV（开发）|
|feature|开发新功能|UAT（用户验收测试）|
|test|测试分支|FAT（功能测试）|
|release|发布分支|UAT|
|hotfix|修复分支|DEV|

### **3. 提交信息规范**

- **feat**：新增功能

- **fix**：修复 Bug

- **docs**：仅文档更新

- **style**：不影响功能的代码格式修改

- **refactor**：既不修复 Bug 也不增加新功能的代码重构

- **perf**：性能优化

- **test**：添加或修改测试代码

- **chore**：构建过程或辅助工具的修改

---

## 💻 示例代码

### **创建功能分支**

```bash
git checkout develop
git checkout -b feature/新功能
```

### **提交信息规范**

```bash
git commit -m "feat: 新增用户登录功能"
```

### **合并到开发分支**

```bash
git checkout develop
git merge feature/新功能
```

---

## ⚠️ 常见误区

- ❌ **直接在 master 上开发**：所有功能开发应基于 `develop` 分支，避免直接修改主分支。

- ❌ **频繁提交**：每次提交都要有明确的功能或修复描述，避免无意义的提交信息。

- ❌ **不统一的分支命名**：保持分支命名的一致性，确保团队成员能够清晰理解分支内容。

---

## 🎯 适用场景

- 大型团队协作开发，尤其适合微服务架构、多个模块的并行开发。

- 项目发布周期较长，且需要多次迭代或修复 bug 的场景。
